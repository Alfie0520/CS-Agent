# 消息链路说明：从用户发消息到收到回复

> 面向 PM 的第一性原理说明，讲清重要概念。

---

## 一、整体流程（5 步）

```
用户发消息 → 微信服务器 → 你的服务器 → 你的服务器 → 微信服务器 → 用户收到回复
```

---

## 二、每一步在做什么

### 1. 用户发消息

用户在微信里给公众号发一条文字，比如「你好」。

---

### 2. 微信服务器收到，并推送到你的服务器

微信收到消息后，会按你事先配置的「服务器地址」把消息 POST 过去。

**重要概念：服务器地址（URL）**

- 你在微信公众平台填的地址，例如：`http://bgtx.work/wx`
- 微信只认这个地址，所有消息都会发到这里
- 和你的 AppID、AppSecret 无关，只和「填的 URL」有关

---

### 3. 你的服务器处理消息

你的服务器收到 XML 格式的消息，会做几件事：

1. **校验签名**：用 Token 验证请求确实来自微信
2. **解析消息**：把 XML 转成结构化数据（谁发的、发了什么）
3. **业务逻辑**：根据内容决定回复什么（例如默认回复、转人工等）
4. **生成回复内容**：例如「您好，我是 AI 客服助手...」

**重要概念：被动回复 vs 客服消息**

- **被动回复**：必须在 5 秒内用 XML 直接返回，适合简单、固定回复
- **客服消息**：先返回「success」，再异步调用接口发消息，适合需要耗时处理的场景（如调用大模型）

当前架构用的是「客服消息」：先快速返回，再异步发回复。

---

### 4. 你的服务器调用微信接口发回复

要发客服消息，必须先拿到 **access_token**，再调用发送接口。

**重要概念：access_token**

- 相当于「临时通行证」，有效期约 2 小时
- 用 AppID + AppSecret 向微信申请
- 申请时，微信会校验：**调用方的 IP 是否在白名单内**
- 白名单未配置 → 拿不到 token → 无法发消息

**重要概念：IP 白名单**

- 微信要求：只有白名单里的 IP 才能申请 access_token
- 目的：防止 AppSecret 泄露后被任意服务器滥用
- 你的服务器 IP 必须加入白名单，才能正常发消息

---

### 5. 微信服务器把回复推送给用户

微信收到你的发送请求后，会把这条消息推送给对应用户，用户就能在微信里看到回复。

---

## 三、MVP 链路小结

| 环节 | 谁在做 | 依赖的配置 |
|------|--------|------------|
| 用户发消息 | 用户 | — |
| 微信推送到你 | 微信 | 服务器 URL、Token |
| 你处理并生成回复 | 你的服务 | 业务逻辑 |
| 你调用接口发回复 | 你的服务 | AppID、AppSecret、IP 白名单 |
| 用户收到回复 | 微信 | — |

---

## 四、一句话总结

**收消息**：微信按你配置的 URL 把消息推给你；**发消息**：你用 AppID + AppSecret 拿 access_token，再调用接口发，且你的服务器 IP 必须在白名单里。
